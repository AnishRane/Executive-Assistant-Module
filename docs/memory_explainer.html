What “hybrid memory” means, in plain words

The Executive Assistant has to remember two different kinds of things. Hard facts — flight numbers, times, confirmation codes — must never get remembered wrong. Soft things — your preferences, who Klein is, which hotel you like in New York — should be allowed to grow and change. Hybrid memory means we use a different store for each.

Structured memory

Postgres tables the module owns. Every column has a type. The database is the source of truth.

Good for: anything that must be exact. Dates, times, flight numbers, prices, who is on a meeting.

Contextual memory

BoringOS’s built-in memory store (the MemoryProvider). The agent writes paragraphs and reads them back when it needs them.

Good for: anything fuzzy. Preferences, vibes, evolving understanding of people and places.

The agent never touches either store directly. It always goes through tools, and every read and write is logged. That is how you can later ask “why did the dossier say that?” and get an answer.

How to read the tables below

Key column

  • PK Primary key. The unique handle for a row. Always called id.
  • FK Foreign key. Points at the id of a row in another table. The target is shown next to it, e.g. → meetings.id.
  • UQ Unique. Not a key relationship, but no two rows can share this value. Useful for things like a Gmail message ID.

Map of how the tables connect

snapshots ──┬── timeline_items ──┬── meetings ──┬── meeting_attendees │ │ ├── thread_excerpts │ │ └── action_items │ │ │ ├── trip_legs ── trips │ │ │ └── ooo_windows │ └── deltas (points back at timeline_items) email_anchors ── points at either meetings or trip_legs (the bookkeeping of which emails the agent has read)

Read it as: a snapshot is the morning's frozen dossier. Each snapshot has many timeline_items. Each timeline item points at one underlying thing — a meeting, a trip leg, or an OOO window. Meetings have attendees, quoted thread bits, and action items. Trip legs roll up into a trip. Deltas are change events tied to a snapshot and a timeline item. Email anchors are the audit trail of which emails were read.

Structured tables Postgres

Module-owned tables, prefixed executive_assistant__. Every row carries a tenant_id; multi-tenant discipline is baked in.

Contextual memory via MemoryProvider

The structured tables above hold the facts. This second store holds the agent’s evolving sense of you. It is not a Postgres table — it is paragraphs the agent writes and reads back through the framework’s memory tools (memory.remember, memory.recall, memory.prime, memory.forget).

Each paragraph lives under a tag (a kind of category name). The agent writes when it sees a pattern; it reads when it composes the dossier.

Memory tagWhat it storesHow it shows up
david.travel.preferences “Leaves 90 minutes before domestic flights, two hours for international. Prefers aisle. Avoids 6 AM departures.” The trip card phrases the airport departure window in his usual cadence.
david.lodging.preferences “Usually stays at the Greenwich Hotel in New York. The Connaught in London. Boutique over chain.” If a hotel is missing from a trip, the agent can quietly note it instead of going silent.
people.<email> “Mark Klein, Acme — VP of partnerships. Pricing is his pet topic. Long emails late at night.” Feeds the attendee bio line on a meeting card when it’s richer than the signature alone.
company.<domain> “Acme — currently in renewal. We’ve been pushing for a multi-year commit; they’ve been resistant.” Lets the meeting brief frame the conversation in its real arc, not as a first meeting.
david.cadence “Reads the dossier between 6:45 and 7:15 most mornings. Rarely opens trip cards until the day before departure.” Helps the agent decide what to elevate vs. tuck into a drilldown.
david.voice “Prefers understatement. No superlatives. Treats logistics as private; agendas as shared.” Tunes the tone of the agent’s written prose.

Why not put all of this in tables?

Because we don’t know in advance what columns the soft stuff needs. If we tried to schematise “preferences,” we’d invent five columns, fill two of them, and quietly stop using the table. Free-form paragraphs let the agent grow its understanding without redesigning the database every time.

Why not put all of it in contextual memory?

Because contextual memory can drift on recall. A flight number is not something we want the agent to “remember” — it is something we want the agent to look up. Tables enforce that.

How one morning uses both stores

The 6 AM job runs. The agent reads the calendar and the in-scope emails. It reconciles each trip leg into trip_legs.current_state, writes a brief for each meeting into meetings.brief, picks the verbatim excerpts to surface, and composes the day’s snapshots.narrative_brief. While it composes, it pulls up the relevant contextual memories — what it knows about Klein, about Acme, about your travel cadence — so the prose sounds like a person who knows you. The structured rows are the facts. The contextual recall is the voice.

Through the day, changes land in deltas, source emails accumulate into trip_legs.source_message_ids, and — when a pattern repeats enough to be worth keeping — the agent writes a new contextual memory. Tomorrow’s dossier sounds a little more like you than today’s did.

Illustrative artifact prepared during brainstorming. Names, shapes, and attributes are proposals — none of this exists in the codebase yet. The point of this document is to make “hybrid memory” a thing you can point at and disagree with, before we lock the design.
