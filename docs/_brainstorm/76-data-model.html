Where each kind lives, and what changes

Five sections, one read. The taxonomy itself, then which existing table holds which kind, then the one schema change needed, then how a single gcal event flows through, then six sample rows showing what kind they'd land in.

1The five kinds

A flat enum. Every item the user sees on /executive is exactly one of these.

Meeting
A work meeting
Has a video link OR a keyword like meeting / catchup / sync / 1:1. The agent treats it strategically.
Example: Q3 board review · 9am · Zoom
Event
A social or personal commitment
Noun-phrase title, usually has a location, often involves other people. Lunches, dinners, birthdays, kids' events.
Example: Lunch with Sarah · 1pm · Quince
Task
A solo errand or focus block
Imperative-verb title. "Pick up X." "Review Y." "Sign Z." Could be on the move or at the desk.
Example: Pick up dry cleaning · 3:20pm
Trip
A multi-day travel envelope
Wraps legs (flight, hotel, ground). Already its own table, populated by the ea-travel-agent.
Example: NYC · Wed–Fri · UA 416 + The Mark
OOO
An unavailability window
A range when the user is marked out. Auto-decline applies. Already its own table.
Example: Fri 19 May · personal day

2Where each kind comes from

The source table for each kind. Trip and OOO already live in their own tables. The other three all live in the executive_assistant__meetings table and are distinguished by a new kind column.

Kind Source table How is the kind known?
Meeting executive_assistant__meetings kind = 'meeting' — set by the rule at sync time.
Event executive_assistant__meetings kind = 'event' — set by the rule (or LLM) at sync time.
Task executive_assistant__meetings kind = 'task' — set by the rule (or LLM) at sync time.
Trip executive_assistant__trips → any row in this table is kind = trip. No column needed.
OOO executive_assistant__ooo_windows → any row in this table is kind = ooo. No column needed.

4How a single gcal event flows in

From Google sync to typed card. The kind is decided once and persisted.

1
Google sync
Calendar API delivers a new or updated event. Row is upserted into __meetings.
2
Rule classifier
In-memory function inspects title, conference_link, attendees. Outputs {kind, confidence}.
3
Write kind
If kind_locked is true, skip. Else write kind. If confidence low, write 'untagged' + queue LLM tagger.
4
LLM tagger (async)
For untagged rows: light prompt with rule context + user memory ("user prefers X as Y"). Writes final kind.
5
UI card
meetings.list includes kind. The web dispatcher picks the right card component.

6User override + learning

When the user re-tags, the row is locked AND a memory note is written so the LLM tagger learns the preference over time.

Task · "Dentist"
→
Event
User right-clicks, picks Event. We set kind='event', kind_locked=true, and write memory: "user prefers Dentist-style as event".
Untagged · "Maya's recital"
→
Event
LLM tagger reads the new memory + the rule context, decides event, writes the column. No user action needed.
Event · "Dentist" (next month)
→
Event
Rule still says task, but LLM tagger reads the memory and overrides. Future dentists match the user's mental model automatically.
