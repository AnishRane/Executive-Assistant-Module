Framework change · refactor/google-oauth-token-access

getConnectorToken — direct OAuth token access for modules

A small change to @boringos/module-sdk and @boringos/core that lets modules (EA, CRM, …) call Google APIs directly with a fresh access token, instead of going through invoke("google.calendar.list_events"). Commit 8390b51, branch refactor/google-oauth-token-access (not merged to main yet).

1The problem

Before the change, the only path for a module to hit Google's API was string dispatch through the tool registry. The token, refresh logic, and provider client all sat behind one wrapper tool per endpoint. That works, but it makes the module brittle.

String coupling

Caller does invoke("google.calendar.list_events", ...). If the Google module renames or restructures the tool, every consumer breaks at runtime, not compile time. There's no type contract.

Wrapper sprawl

Every Google endpoint a module wants needs its own google.* wrapper tool. EA needed sync-token-based incremental polling (v0.4.19) — the wrapper would need to be updated to support that query shape. Faster to call Google directly.

Token logic scattered

Each google.* tool repeats the refresh-if-expiring dance internally. If the threshold or persistence policy changes, you edit it in N places.

2Before vs. after

Same EA goal: "list this tenant's upcoming Google Calendar events." Two different paths.

Before · tool-registry indirection
1 EA calls invoke("google.calendar.list_events", {timeMin, timeMax}).
2 Tool registry resolves the string, dispatches to the Google module's wrapper tool.
3 Wrapper loads creds from DB, refreshes the access token if it's expiring.
4 Wrapper instantiates CalendarClient, calls the Google endpoint.
5 Wrapper unwraps the Google response into its own shape and returns to EA.
6 EA reads the wrapped shape — coupled to the wrapper's contract.
After · direct API call with a fresh token
1 EA calls deps.getConnectorToken("google", tenantId) at sync time.
2 Core's getConnectorTokenForTenant loads creds, refreshes if <60 s to expiry, persists the new token, returns {accessToken}.
3 EA calls Google's REST API directly: fetch(CALENDAR_API + params, { headers: { Authorization: \`Bearer \${accessToken}\` } }).
4 EA parses Google's native response shape — no wrapper in between.
Net effect: two fewer hops, no string dispatch, and the only thing the Google module owes EA is a token. EA owns the URL shape, the query string, the response parsing, the sync-token state — all the things that change as EA's needs evolve.

3What changed in code

Three files changed materially. The SDK gained a typed field; the host injected it; the Google module exposed the underlying function.

3b · core/modules/google.ts: the implementation

A pure function exported from the Google module. Loads creds, checks expiry (60 s buffer), refreshes if needed, persists the new token, returns {accessToken} or null.

New · packages/@boringos/core/src/modules/google.ts
export async function getConnectorTokenForTenant(
  db: Db,
  kind: "google" | "slack",
  tenantId: string,
): Promise<{ accessToken: string } | null> {
  if (kind !== "google") return null;

  const creds = await loadGoogleCreds(db, tenantId);
  if (!creds) return null;

  // Proactive refresh: avoid the round-trip 401 + retry dance.
  const expiresAt = creds.rawCredentials.expiresAt as string | undefined;
  const expiresSoon = expiresAt
    ? Date.now() > new Date(expiresAt).getTime() - 60_000
    : false;

  if (expiresSoon && creds.refreshToken) {
    const refreshed = await refreshOAuthToken("google", creds.refreshToken);
    if (refreshed) {
      await db.update(connectors)
        .set({ credentials: { ...creds.rawCredentials, accessToken: refreshed.accessToken,
                              expiresAt: refreshed.expiresAt } })
        .where(...)
        .catch(() => {}); // best-effort persist
      return { accessToken: refreshed.accessToken };
    }
  }
  return { accessToken: creds.accessToken };
}

5How EA consumes it

The consumer side already shipped in EA v0.4.7. Two files:

5a · EA reads the dep optimistically

Bracket-notation access so EA compiles against older versions of module-sdk (where the field doesn't exist yet) and still picks it up when the framework is updated.

packages/server/src/module.ts
// Read the field with a double-cast so EA's published module-sdk
// (which doesn't have the field) and the host's worktree (which does)
// both compile.
const getConnectorToken = ((factoryDeps as unknown as Record<string, unknown>)["getConnectorToken"]
  as EaDeps["getConnectorToken"] | undefined) ?? null;

const deps: EaDeps = { db, getEventBus, getRealtimeBus, toolRegistry, getConnectorToken };

5b · The sync handler uses the token directly

packages/server/src/tools/calendar.ts
const tokenResult = await deps.getConnectorToken?.("google", ctx.tenantId) ?? null;
if (!tokenResult) {
  return { ok: true, result: { data: { synced: 0, skipped: true, reason: "google_not_connected" } } };
}

const res = await fetch(`${CALENDAR_API}?${params}`, {
  headers: { Authorization: `Bearer ${tokenResult.accessToken}` },
});
// EA parses Google's native response — no wrapper.
Fail-soft: if the field isn't injected yet (e.g., the framework branch isn't merged on the host), deps.getConnectorToken?.() short-circuits to null, EA returns { skipped: true, reason: "google_not_connected" }, and the day brief still works (from whatever's already in the DB).
