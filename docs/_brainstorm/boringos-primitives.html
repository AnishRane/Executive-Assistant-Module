Modules, Plugins, Connectors — what they are and how they fit

Three nouns the framework throws at you, plus one of them (connector) means three different things depending on context. This doc unpacks each, draws the boundaries, and walks an end-to-end "I want to build a module that needs Slack" scenario so the seams become obvious.

1At a glance

Three primitives, three jobs. They're not alternatives — they coexist in one host and lean on each other. Tap each card to learn more in §2–§4.

Module
app.module()

A bundle of skills + tools the agent can read and call.

  • Owns DB schema (prefix <id>__)
  • Ships agent tools, skills, UI screens
  • Optional OAuth, webhooks, default workflows / agents / routines
  • Per-tenant install/uninstall via .hebbsmod upload
  • Three roles: connector, module, hybrid (UI hint only)
Plugin
app.plugin()

Background jobs + inbound webhooks. No agent surface.

  • Cron jobs with persistent KV state
  • Webhooks mounted at /webhooks/plugins/<name>/<event>
  • Per-tenant config stored in DB
  • No tools, no schema, no UI, no agent skills
  • Always-on (no install/uninstall lifecycle)
Connector
three meanings

An overloaded word — pin which layer you mean.

  • (a) A row in the connectors table = OAuth grant for tenant X to provider Y
  • (b) A npm package like @boringos/connector-google = the HTTP client to talk to Google
  • (c) A Module with kind: "connector" = the agent-facing wrapper that exposes tools

2Module — the universal component

Everything a tenant can install and uninstall as a unit is a Module. It's the only primitive that touches the agent: tools, skills, and UI all live here. Three "roles" play different mixes of the same manifest.

Same manifest, three roles

RoleWhat it ownsTypical fieldsReal examples
Connector kind:"connector" A 3rd-party integration (Gmail, Slack) oauth, tools, webhooks, events core google, slack
Module kind:"module" Business logic + data over other Modules schema, dependsOn, tools, provides EA, Hebbs CRM, triage
Hybrid kind:"hybrid" Owns data and brokers a 3rd-party both columns above combined Stripe Billing (hypothetical)
"Same shape, different role" is the design philosophy. A Module that owns no DB schema and only brokers OAuth is a connector; one that owns schema is a "module" in the narrower sense; one that does both is hybrid. The framework treats all three identically — install, dispatch, lifecycle. The kind field is just a UI hint (group under Connectors vs. Apps in the shell).

What a Module ships (fields the manifest accepts)

  • tools — Zod-typed callables dispatched at /api/tools/<id>.<name>. Agent calls these.
  • skills — markdown injected into the agent's prompt under ## Skills.
  • schema — Drizzle migrations. Tables MUST be prefixed <id>__.
  • workflows / agents / routines — seeded on install; editable rows the tenant owns afterward.
  • webhooks — inbound HTTP at /api/webhooks/<id>/<event>. The Module owns signature verification.
  • oauth — declarative OAuth config. Status: declared in SDK, built-ins still use the legacy connectors table.
  • ui — React screens / panels the shell mounts.
  • lifecycle — onInstall, onUninstall, onTenantCreate hooks.
  • dependsOn / provides — capability resolution across modules.

How modules get registered

Two paths, both end at the same app.module() registration:

  • Static, at boot. Built-ins call app.module(myModule) before app.listen(). The 11 built-ins (framework, memory, drive, workflow, inbox, slack, google, copilot, triage, inbox-triage, inbox-replier) live in core/src/modules/.
  • Runtime upload. Third-party Modules ship as a single .hebbsmod bundle. Admin drops it on the Apps screen → host extracts, signature-checks, dynamic-imports the bundle, calls the same registration path on a live process. No restart, no npm publish, no source checkout.

3Plugin — the lightweight cousin

Plugins are a separate primitive, not a kind of Module. They run cron jobs and receive webhooks, but they don't expose agent tools, don't own schema, don't have a UI, and there's no per-tenant install/uninstall ceremony.

When you'd reach for a plugin instead of a module

NeedModulePlugin
Expose tools the agent can call ✅ tools[] ❌ Not supported
Own DB tables / schema ✅ schema with <id>__ prefix ❌ Per-tenant KV state store only
Ship UI screens / panels ✅ ui field ❌
Run a cron job ✅ routines[] ✅ jobs[] with cron schedule
Receive inbound webhooks ✅ /api/webhooks/<id>/<event> ✅ /webhooks/plugins/<name>/<event>
Tenant install lifecycle ✅ module_installs row + onInstall hook ❌ Always-on; just config + state per tenant
Distribution format .hebbsmod bundle uploaded via admin UI Code-only; registered via app.plugin() at host boot
Mental model: Plugin is "I want to run a CRON job and receive webhooks, that's it." If you ever need an agent to call into your code or you need your own tables, you're really writing a Module and should drop the plugin form. The shipping GitHub plugin is the reference example — it syncs GitHub issues into BoringOS tasks every 15 minutes and posts incoming webhook events.

4"Connector" — the word that means three things

When somebody says "connector" in BoringOS, they could mean any of three layers. The framework's docs use the word for all of them. Pin which one you mean before designing.

LayerWhat it isWhere it lives
aConnector grant A single row in the connectors table = "tenant X has OAuth tokens for provider Y (google or slack)". Stores access + refresh token, expiry, scopes. @boringos/db · table connectors
bConnector package An npm package that wraps a 3rd-party provider's HTTP API. Pure transport — no Module manifest, no agent surface. Other code imports and uses the client. @boringos/connector-google (GmailClient + CalendarClient)
@boringos/connector-slack
cConnector Module A Module manifest with kind: "connector". Imports the connector package's clients, wraps them in Tool definitions, declares OAuth scopes, exposes them to agents. core/src/modules/google.ts
core/src/modules/slack.ts

Worked example — how the three layers compose

Tenant connects
User clicks "Connect Google" in BoringOS. OAuth dance lands an access token + refresh token. A row is inserted into connectors with (tenant_id, kind:"google", credentials:{...}).
Layer (a) — connector grant
Module wants data
EA's calendar.sync_window tool runs. It needs to call Google. It loads the token from connectors (or via the new getConnectorToken helper).
Layer (a) read
HTTP call
EA uses the token to call Google REST directly, or — in the older path — invokes google.calendar.list_events (a tool exposed by the Connector Module). That tool uses CalendarClient from the connector package.
Layer (b) + (c)
Result back
Google returns events. EA writes them to its executive_assistant__meetings table (its own schema, layer-(c) doesn't see this). The agent later reads them via EA's tools.

5Walkthrough · "I want to build a Module that needs Slack"

You're an open-source contributor. You want to write a Module — let's call it deal-tracker — that watches a Slack channel for keywords and creates tasks. Here's how the pieces fit, today.

1 · Scaffold
Generate a Module skeleton with npx create-boringos (or copy examples/quickstart). You get module.json, src/module.ts, a SKILL.md, a migrations.ts, and a package.json. Declare your manifest in src/module.ts.
2 · Declare deps
In your Module manifest: dependsOn: [{ moduleId: "slack" }]. The framework guarantees the slack Module is loaded before yours. Your tools can now call slack.send_message, slack.list_channels, etc.
3 · OAuth
You don't implement OAuth. The Slack Connector Module already handles "tenant X is connected to Slack" — your Module just calls Slack tools and the connector resolves the token from the connectors row at call time. Your tenant has to click "Connect Slack" once in BoringOS's settings UI.
4 · React to Slack events
Today, the path is rough. Slack pushes events to a URL you must register in Slack's app settings. If your Module declares webhooks: [{ event: "messages", verify, handler }], the framework mounts a route at /api/webhooks/deal-tracker/messages. You still have to: manual expose HTTPS in dev (ngrok), manual register the URL in Slack's app config, manual implement HMAC signature verification, manual de-dupe replays.
(This is exactly the friction your brainstorm targets.)
5 · Local test
Two options:
  • Unit-test your tools against a mock Slack client. Vitest + @boringos/agent's test dispatcher. No Slack needed.
  • Run a dev BoringOS host. pnpm dev in the framework repo (embedded Postgres boots automatically). Set HEBBS_DEV_MODULES=true to accept unsigned bundles. Pack with pnpm exec pack-hebbsmod, drop the file on the Apps screen, install for your dev tenant, watch logs.
6 · Ship
Bump version in module.json. Re-pack. Sign with npx sign-hebbsmod if shipping to a production host. Upload via admin UI or POST /api/admin/modules/upload. The framework persists every version in module_packages so tenants pin to specific versions.
What this means for testability: a contributor can run the entire framework locally with one command, build their Module as a .hebbsmod, and exercise the full install → tool dispatch → uninstall loop without any cloud dependencies. The friction is concentrated in step 4 — receiving provider events. That's the gap the brainstorm is sizing up.

6Where the friction lives today

The pieces above mostly work for one-way "module calls provider" flows. Inbound provider events are where the abstraction stops pulling its weight. Concrete pain points:

1

HTTPS exposure in dev. Slack, Google, Stripe etc. all push to public URLs. Each contributor has to run ngrok / Cloudflare Tunnel / similar themselves. The framework doesn't help.

2

Provider-side registration is manual. Module author has to log into Slack's developer console (or Google's Cloud Console) and paste a URL. Re-doing this per dev / staging / prod environment is per-environment friction.

3

Webhook lifecycle is the Module's problem. Google Calendar channels expire after 7 days; you need a cron job to renew them. The framework provides no helper.

4

Signature verification is repeated per Module. Slack HMAC, Stripe HMAC, GitHub HMAC — each Module reimplements the same dance with the same per-tenant secrets. The framework's webhooks[].verify is a hook, not a helper.

5

No push path for the Google connector. The shipping connector-google + google Module exposes pull-only tools (list_events, list_emails). EA had to write polling tools because there was no way to subscribe to push notifications via the framework.

6

EventBus is in-process only. The framework's ConnectorEvent bus is fine for "Slack module → CRM module" routing inside one host, but it doesn't help the "Slack → BoringOS" direction. That's exactly the inbound-event problem.

7One-line summaries to remember

Module The universal component. Agent-facing. Tools, skills, schema, UI, lifecycle, webhooks, OAuth. Three roles (connector / module / hybrid) — same shape.
Plugin Lightweight cousin. Cron jobs + inbound webhooks + KV state. No agent surface. Use when there's no agent involvement at all.
Connector Overloaded word. Either an OAuth grant row, an npm client package, or a Module with kind:"connector". Disambiguate before designing.
The gap Inbound provider events. No unified primitive for "wake my module when something happens at the provider." Module authors handle HTTPS exposure, provider registration, channel renewal, signature verification, and replay protection themselves.
