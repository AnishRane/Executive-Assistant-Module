01Calendar & meetings

Google Calendar is the canonical source of structured events. Pulled on a 15-minute cadence; every write checks for changes and records what shifted.

Calendar sync Working

Pulls events from Google Calendar and upserts them locally so all downstream features have structured data.

How
Routine EA Sync Calendar every 15 min → workflow → meetings.upsert_from_calendar.
Source
Google Calendar via @boringos/connector-google.
Lands in
executive_assistant__meetings + __meeting_attendees.
Surface
/executive timeline and /executive/meetings/:id.
Note
Starts paused. Auto-activates when connector.google.connected fires.

Meeting change detection Working

When a calendar event's title or time shifts vs. what we have, the change is recorded synchronously on the same write.

How
Comparison inside meetings.upsert_from_calendar calls detectMeetingDelta in the same request.
Lands in
__deltas (kind rescheduled / meeting_updated); also recomputes __conflicts.
Surface
Inline change badges on Today + Meeting Detail.

Per-meeting brief, attendees, excerpts, action items Working

Each meeting on the dossier has a one-paragraph brief (≤60 words), attendees with bios, up to two verbatim email excerpts, and extracted open items.

How
executive-assistant agent during morning compose; calls meetings.set_brief, thread_excerpts.create, action_items.create.
Source
Meeting + attendee rows · email_anchors for Gmail content · MemoryProvider for people.<email> / company.<domain>.
Surface
/executive/meetings/:id.

Action item dismiss Working

User can dismiss an action item; the dismissal is captured as a feedback signal so the agent learns when it over-extracts.

How
action_items.dismiss flips status to ignored + writes dismissed_action signal.
Surface
Meeting Detail.

02Travel

No vendor-locked regex parsers. The ea-travel-lens agent reads any email that BoringOS triage already classified, decides if it's travel, and extracts structured leg data.

Travel email ingestion Working

Wakes per email after triage classification; uses an LLM to decide travel vs. not and to extract leg fields. No provider-specific parsers.

How
Workflow EA Travel Lens on triage fires on triage.classified → task for ea-travel-lens agent → trip_legs.reconcile_from_email.
Source
Gmail inbox items after BoringOS triage. Idempotent on metadata.eaTravelLens.processedAt.
Lands in
__trips + __trip_legs (and an anchor row in __email_anchors).
Surface
/executive/trips/:id plus active trips on the Today timeline.

PNR deduplication Working

Two emails for the same booking don't make two legs. Confirmation code is the durable identity.

How
Partial unique index on (trip_id, kind, confirmation_code) WHERE confirmation_code IS NOT NULL. Reconciler merges field-by-field, appending message IDs.

Trip grouping (fuzzy match) Working

A flight, hotel, and ground transfer for the same trip cluster onto one trip record.

How
findOrCreateTrip: same destination (case-insensitive) + ±7 days date overlap.
Note
±7-day window may create separate trips for multi-city itineraries with wide gaps.

Leg change detection Working

A flight time change confirmation reflows the leg and surfaces a delta.

How
Reconciler compares before/after times; calls detectTripLegDelta (kind flight_delayed or trip_leg_updated).

Live flight status polling Not in v0.1

Would call a flight-status API in real time to catch delays the airline emails late.

Why not
Out of scope by design. All leg updates currently come from emails.

03Email anchoring

An email enters the agent's context only if it has an anchor row tying it to a known entity (meeting thread or travel leg). There is no “read all mail” path.

Meeting thread scan (bulk, deterministic) Working

How
Routine EA Scan Meeting Threads (15 min) calls email_anchors.scan_meetings. SQL join on thread_id = meetings.gmail_thread_id. No LLM, no regex.
Lands in
__email_anchors with anchor_kind = meeting_invite.

Travel anchor binding (agent path) Working

How
After ea-travel-lens reconciles a leg, it calls email_anchors.bind with travel_confirmation / travel_update.

Scope enforcement Working by design

Architectural constraint, not a feature. The schema and workflows make “read all email” impossible without a code change.

04Out-of-office windows

OOO sync from calendar Working

How
Called alongside meeting sync. ooo.upsert_from_calendar uses SELECT-then-UPDATE (the unique index is partial).
Lands in
__ooo_windows.
Surface
Timeline items on Today view.

OOO change detection Working

How
detectOooDelta writes a ooo_updated delta when start/end shifts.

05Daily dossier composition

The morning narrative is one paragraph under 80 words in David's voice. It reads the day to him, not at him.

Morning narrative brief Working

How
Routine EA Morning Compose at 06:00 daily → task for executive-assistant agent → snapshots.create.
Source
__meetings, __ooo_windows, __trips + MemoryProvider for david.voice / david.cadence.
Lands in
__snapshots (status composed; prior same-date snapshot becomes superseded).
Surface
/executive — the headline paragraph.

Timeline construction Working

How
Agent calls timeline_items.create for every meeting, leg, OOO. Items starting before 07:30 are elevated with reason imminent.
Lands in
__timeline_items.

Pre-06:00 entity fallback Working

If you open /executive before the morning agent has run, the page renders directly from the underlying entity tables — never blank.

Empty-day handling Working

No meetings, no travel, no OOO still produces a snapshot. The narrative may be a single sentence (“A quiet Tuesday.”).

Snapshot history (90 days) Working

Surface
/executive/history. Each row links back to the dossier as it appeared.

Weather block (expanded) Working

Two-line block above the narrative. Line 1: location · granular condition · high/low (feels-like if diverged). Line 2: conditional chips for precip, wind gusts, UV, sunrise/sunset. Fetched by the morning agent so location can be travel-aware.

How
Agent resolves location during morning compose, calls weather.fetch_for_date. Tool hits Open-Meteo's daily endpoint and upserts a row. UI reads via weather.get_for_snapshot; block is hidden when no row exists.
Provider
Open-Meteo · no API key required · daily forecast endpoint.
Captured
WMO weather code (mapped two ways: 6-bucket enum and granular WMO label like “Heavy rain”, “Thunderstorm with slight hail”) · dry-bulb high/low · apparent (feels-like) high/low · precipitation sum (mm) + probability · wind speed + gusts (km/h) · UV index · sunrise · sunset.
Lands in
__weather_snapshots (incl. raw_payload for re-derivation).
Feedback
Overflow menu fires weather_dismissed, weather_too_detailed, or weather_too_terse signals. Daily / weekly reflection tunes david.weather.preferences.
Narrative
Agent mentions weather when feels-like diverges ≥5°C, gusts ≥40 km/h, precip ≥40% or ≥1 mm, UV ≥8, or any heavy/freezing condition. Mild + clear + calm stays silent — chips already carry the numbers.

06Live deltas

If anything shifted since the snapshot was composed, it surfaces as a delta — not silently re-rendered.

Delta recording Working

How
Synchronous, write-triggered: detectMeetingDelta, detectTripLegDelta, detectOooDelta in delta_engine.ts. Wrapped in try/catch — a delta failure never rolls back the parent write.
Lands in
__deltas with delta_kind, summary, meta (before/after), occurred_at.

Acknowledge & mark-unhelpful Working

How
deltas.acknowledge stamps acknowledged_at. deltas.mark_unhelpful writes a marked_unhelpful feedback signal.

60-second UI refresh Working

React Query refetchInterval: 60_000 on timeline / deltas / conflicts / feedback signals. Worst case is ~16 minutes from a calendar move to it appearing (15-min routine + 1-min poll).

Note
Originally planned as SSE push — shipped as polling. No SSE path exists in the web code.

07Conflicts

Pairwise overlap detection Working

How
After any delta write, recomputeConflictsForSnapshot does an O(n²) pair scan. CHECK constraint enforces canonical ordering (a_id < b_id) so inverse duplicates can't exist.
Lands in
__conflicts. Pairs that previously overlapped but no longer do are auto_resolved.

User resolution (pick a winner) Working

How
conflicts.resolve validates the chosen id, sets resolution_status = resolved_by_user, writes conflict_resolved feedback signal with overlapMinutes.

Draft suggestion (“ask X to swap”) Not in v0.1

Recording the choice works today. Composing a reply draft for the “loser” party arrives in v0.2 once we wire outbound drafts.

08Feedback loop

Every meaningful user interaction is captured as a queryable, traceable signal. The agent uses these to refine its voice — and the loop is visible at /executive/memory.

Brief edits → edited_brief Working

When David rewrites an agent-composed brief, the diff (before, after, word delta, sentence delta, prose summary) is captured.

How
meetings.set_brief distinguishes via ctx.agentId: absent = user edit, signal written; present = agent write, no signal.

Action dismiss · delta unhelpful · conflict resolve Working

Three more user-driven signal kinds, all recorded automatically on the underlying tool calls.

Telemetry (opened_card, acknowledged_delta) Partial

Schema slots and the feedback_signals.record tool both exist. The UI doesn't yet fire opened_card on every card open — wiring is pending.

edited_attendee_bio · pinned_thread Not built

Schema slots exist; no UI affordance currently triggers them.

Daily reflection Working

End of every morning run: pull last ~100 signals; commit a memory update when a pattern holds across 3+ rows.

Source
__feedback_signals + MemoryProvider read/write.
Lands in
MemoryProvider (external to module DB).

Weekly deep reflection Working

Mondays 06:30: 7 days of signals aggregated by kind to detect trends invisible day-to-day (e.g., recurring dismissals from one company).

How
Routine EA Weekly Reflection → task for the same executive-assistant agent with title Weekly memory reflection.

10UI surfaces

/executive — Today Working

Narrative brief + ordered timeline. Two render paths: snapshot-driven (canonical) or live-entity fallback (pre-06:00). Polls every 60s.

/executive/meetings/:id — Meeting Detail Working

Attendees · editable brief · verbatim excerpts · action items (dismissible) · per-item deltas · conflict resolve affordance.

/executive/trips/:id — Trip Detail Working

All legs in order with confirmation codes, times, locations.

/executive/history — Snapshot History Working

Last 90 days of composed dossiers; each row links back to that day's view.

/executive/memory — Memory & Evidence Partial

Feedback signal stream (last 50) is live and polling. Contextual memory tag display is not yet wired — placeholder copy only.

/executive/settings — Settings Working Read-only

Lists routines, memory tags, dependencies. No pause/resume/forget actions — those live in the BoringOS admin UI for now.

11External dependencies

Google Calendar + Gmail Wired

Via @boringos/connector-google. Required for calendar sync, OOO sync, meeting thread anchoring, and travel email ingestion. Routines auto-activate on connector.google.connected.

BoringOS Triage Wired

Built-in framework agent. Classifies inbox items; the triage.classified event is what wakes ea-travel-lens.

MemoryProvider Wired

Framework memory.recall / remember / forget. Used by daily + weekly reflection. The morning compose degrades gracefully if it's unavailable — structured data still produces a snapshot.

Agent runtime (Claude) Wired

Agent seeding is conditional: lifecycle.ts skips seeding (and logs a warning) if no Claude runtime or root agent exists for the tenant. Workflows and routines are still seeded.

12What is not here

Explicitly verified absent from the codebase. This is the list to cite when someone asks “does EA also do X?”

Not implemented in v0.1

  • News & market data. Out of scope for Personal AI Stack Phase 1.
  • Commute & traffic. No Maps / Waze / transit integration.
  • Outbound email. Read-only on Gmail; no send path.
  • Reply drafts. Conflict resolution records the choice but doesn't compose a draft. v0.2.
  • Calendar edits / auto-reschedule. All calendar I/O is inbound.
  • Slack / team messaging. No connector, no surface.
  • SMS / push. No Twilio, no push service. In-app only at /executive.
  • Email dossier delivery. Deliberate divergence from the master plan — in-app only by design.
  • Voice interface. No STT/TTS, no voice affordance.
  • Live flight status polling. Leg updates come from emails only.
  • SSE realtime push. Replaced by 60s polling in the shipped build.
  • Pre-meeting briefs from prior meetings. Waiting on the Meetings Assistant (Week 4 of master plan); slots show “no prior meeting on file” until then.
  • Mobile-optimised UI. Desktop-first; container caps at 940px.
  • Buffer-thin back-to-back warnings. Conflict engine only fires on actual overlap.
  • Direct settings mutation. Pause/resume/forget live in the BoringOS admin UI; v0.2 may expose them here.
  • Memory content shown in UI. Feedback stream is live; the tag values themselves (david.voice etc.) are not yet displayed.
