1. The problem with polling — and why it feels slow

Right now the dossier page works like this: every 60 seconds, the browser asks the server, "Anything new?" The server answers "No, same as last time" — about 95% of the time. Once in a while it answers "Yes, here's a meeting that moved."

time → Browser ─── "anything new?" ──► Server : "no" ←── 60 s of waiting ── Browser ─── "anything new?" ──► Server : "no" ←── 60 s of waiting ── Browser ─── "anything new?" ──► Server : "no" ←── 60 s of waiting ── Browser ─── "anything new?" ──► Server : YES — meeting moved (this is the only useful call)

Two costs:

  • Latency. Worst case, you wait 60 seconds for a change you'd want to see in 2 seconds. For someone managing back-to-back meetings, that's an eternity.
  • Waste. The other 95% of those requests do nothing useful — they just keep checking. The server has to handle them; the browser uses up battery; the network gets noise.

Polling is asking "Are we there yet?" over and over on a road trip. There's a better way: have the driver tell you when we arrive.

2. SSE — the phone line that stays open

Server-Sent Events (SSE) flips the conversation. The browser calls the server once, says "keep me on the line", and then waits. The server holds the line open and pushes messages down it whenever something happens. No more re-dialling every minute.

time → Browser ─── "subscribe to /events" ──► Server ↓ connection stays open ─ (silence — nothing's happening) ─ ─ (silence) ────────────────────── ─ (silence) ────────────────────── ↓ something changes ◄─── "meeting moved: id=abc, new start=15:30" ─── ─ (silence) ────────────────────── ↓ something else changes ◄─── "conflict detected: 30 min overlap" ──── ─ (silence) ──────────────────────

The connection isn't doing anything most of the time — and that's the point. The browser isn't checking. The server isn't answering "no". Both are idle until there's actually news. When there is, it arrives within milliseconds rather than up-to-60-seconds.

Under the hood SSE is just a regular HTTP response that never finishes. The server writes data: … lines to it whenever it has something to say, and the browser reads them as they arrive.

"Server-Sent" means one-way. The server can push to the browser, but the browser can't push to the server through the same channel. The browser still uses normal HTTP requests for that (when you click "Resolve conflict", that's a regular POST). SSE is purely for "here's something new" notifications.

3. "Bus" — one phone line, many listeners

So far we've described one browser, one server. But what if multiple pages are open? What if the admin dashboard wants to see the same events? What if a future "Memory" page wants to react to the same conflicts?

Instead of every page having its own private phone line for every event type, the server keeps one shared channel — the bus — and any browser that's interested subscribes to it. Inside the server, components publish events to the bus, and the bus broadcasts to every connected subscriber.

Publishers Subscribers ╔══════════╗ ┌──────────────┐ publish ───► ║ ║ ──► ┌──────────────────┐ │ EA module │ ║ ║ │ /executive page │ └──────────────┘ ║ ║ └──────────────────┘ ║ ║ ┌──────────────┐ publish ───► ║ BUS ║ ──► ┌──────────────────┐ │ Triage agent │ ║ ║ │ /memory page │ └──────────────┘ ║ ║ └──────────────────┘ ║ ║ ┌──────────────┐ publish ───► ║ ║ ──► ┌──────────────────┐ │ Workflow run │ ║ ║ │ Admin dashboard │ └──────────────┘ ╚══════════╝ └──────────────────┘ (server-side) (in browsers)

The bus is the metaphor that makes this useful — pages don't have to know who's publishing, publishers don't have to know who's listening. Add a new publisher (a new module) and existing pages see its events automatically. Add a new subscriber (a new page) and it picks up everything the bus is carrying without any wiring change.

Pages can also filter — "I only care about events with kind=meeting_updated". The bus delivers everything; the page picks what's relevant.

4. SSE vs polling vs WebSockets — when does each fit?

Polling SSE WebSockets
How it talks Browser keeps asking Server keeps a one-way line open Two-way line open
Latency for changes Up to one polling interval (often 60 s) Milliseconds Milliseconds
Wasted "no change" traffic A lot — every poll, every tab None None
Complexity Trivial — any HTTP works Simple — plain HTTP + EventSource API More moving parts; needs WS server, framing, ping/pong
Auto-reconnects on drop? — Yes, built into browser You write the reconnect logic
Browser → server messages Yes — normal requests No on the SSE channel itself; use a normal POST for actions Yes — same channel
Best for Occasional state checks Notifications, dashboards, dossiers, live tickers Chat, multiplayer games, collaborative editors

For the dossier, SSE is the right tool. The browser doesn't need to push anything through the live channel — clicks and edits already use regular POSTs to existing tools. We just want push-from-server when meetings change, conflicts appear, or briefs refresh. SSE does exactly that, simpler than WebSockets, faster than polling.

5. How BoringOS uses the SSE bus today

The host already ships an SSE bus — it's listed in boringos/CLAUDE.md as one of the standard features. Right now the framework itself uses it for things like agent-run progress ("started", "tool called", "finished"), workflow ticks, and admin-page status. Open the admin run-detail page and watch a run unfold in real time — that's SSE under the hood.

Anatomy of the bus, at a high level:

CONCEPT · 01 The SSE bus inside BoringOS
  1. The host exposes GET /api/events (or similar) — that's the SSE endpoint a browser subscribes to.
  2. The browser's EventSource API connects. The host's auth middleware checks identity. The connection stays open.
  3. Server-side, any module can publish through a typed helper — usually eventBus.publish({ kind, tenantId, payload }).
  4. The host broadcasts each event to every connected subscriber on the same tenant. Other tenants don't see it.
  5. If the connection drops (network blip, sleep), the browser's EventSource auto-reconnects. The server emits a Last-Event-ID hint so the browser can pick up from where it left off.

The EA module gets the event bus injected the same way it gets the database and tool registry — via deps.getEventBus() in the module factory. We just haven't called publish from anywhere yet.

6. What changes if the EA publishes on it

Right now the dossier page polls every 60 seconds for snapshots, meetings, conflicts, deltas, weather, week-data. Eight queries, each on its own timer. If a sync detects a conflict, the user sees it on the next 60-second tick.

With the bus wired, the EA publishes events on each meaningful change:

meetings.upsert_from_calendar finishes └─► eventBus.publish({ kind: 'meeting.updated', tenantId, meetingId }) recomputeConflictsForDate writes a new conflict └─► eventBus.publish({ kind: 'conflict.detected', tenantId, conflictId }) meetings.set_brief writes a brief └─► eventBus.publish({ kind: 'brief.changed', tenantId, meetingId }) preferences.set('current_location', ...) └─► eventBus.publish({ kind: 'location.changed', tenantId })

The browser side stops timer-polling and starts listening: 

// hooks/useEaBus.ts (sketch)
useEffect(() => {
  const es = new EventSource("/api/events?topics=ea.*");
  es.addEventListener("conflict.detected", () => {
    queryClient.invalidateQueries({ queryKey: ["ea", "conflicts"] });
  });
  es.addEventListener("meeting.updated", () => {
    queryClient.invalidateQueries({ queryKey: ["ea", "meetings"] });
  });
  return () => es.close();
}, []);

Each event arriving tells the React Query cache to invalidate the matching key. The hook re-fetches once, the UI re-renders. No more 60-second timers; updates show up the moment the server has new data.

Before — polling

8 queries × 60s = 8 round-trips per minute, every minute. Worst-case latency 60 s. Updates feel laggy. Battery drain on the user's device. Server handles a lot of "nothing changed" requests.

After — SSE bus

1 connection open per tab. Server pushes only when something actually changes. Worst-case latency milliseconds. The dossier feels alive — a meeting you just rescheduled in Google appears on the dossier almost as fast as in Google's own UI.

Importantly, this is Leg 2 of the architecture (server → UI). It doesn't solve the Google-Calendar → server problem (that one needs webhooks or fast polling). But it unlocks "instant" everywhere from the server inward. Combined with the sync-token incremental polling we just shipped, end-to-end latency for a calendar edit becomes:

Google Calendar edit │ (up to ~60 s — bounded by 1-min cron) ▼ Server upserts row, fires conflict scan │ (milliseconds — publishes to SSE bus) ▼ Browser receives event, invalidates query │ (milliseconds — refetch is fast) ▼ UI re-renders with the change

7. Try it yourself — a 20-line example

You can taste SSE without any framework. Here's a tiny Node server that streams the current time to anyone who connects, and a one-line HTML page that listens.

Server

import http from "node:http";

http.createServer((req, res) => {
  res.writeHead(200, {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache",
    "Connection": "keep-alive",
  });

  const tick = () => {
    res.write(`data: ${new Date().toISOString()}\n\n`);
  };
  const id = setInterval(tick, 1000);
  req.on("close", () => clearInterval(id));
}).listen(3001);

Browser

<script>
  const es = new EventSource("http://localhost:3001");
  es.onmessage = (e) => console.log("got:", e.data);
</script>

Open the page, look at the console. You'll see one line per second, pushed by the server, with no polling. Close the tab and the server's req.on("close") fires — that's the auto-cleanup. Refresh and the browser reconnects automatically. That's SSE in its entirety.

The BoringOS bus is the same idea, just typed and routed: every event has a kind and a tenantId, the host filters per-subscriber, and subscribers can listen for specific kinds rather than everything.

Where this lives in the EA roadmap. The sync-token polling work we just shipped (v0.4.19) handles Leg 1 — calendar to server. The SSE bus work would handle Leg 2 — server to UI. They're independent; either one shipped on its own is a real improvement. The current cost of NOT having SSE is the 60-second polling lag the dossier still has even though the server has the data already.
