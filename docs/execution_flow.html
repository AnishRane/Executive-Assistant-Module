The cast

Six actors keep showing up across the scenes. Names and code locations matter — keep them in your head.

🏠
BoringOS host Runs the framework, owns the DB, schedules cron, dispatches events, runs LLM tasks.
📦
EA module The .hebbsmod bundle you installed. Schema, tools, skills, lifecycle. packages/server/src/module.ts
🧠
executive-assistant agent The main composer. Wakes on a cron-driven task. Reads SKILL.md, calls tools, writes the dossier.
✈️
ea-travel-agent Specialist email reader. Wakes only on triage.classified events. Turns travel emails into trip_legs.
⏰
The cron Framework-scheduled routines, every 15 min. Fires sync + compose workflows.
🖥️
The /executive UI React app that reads the dossier. packages/web/src/slots/Today.tsx

Scene 1 — The install moment

SCENE · 01 You upload the .hebbsmod. The framework runs migrations, then calls onInstall. Sync (~5s total)

Everything in this scene runs synchronously inside the install transaction. By the time onInstall returns, the tenant has a working EA shell — even though the dossier itself hasn't been composed yet.

  1. Schema migrations
    Creates the executive_assistant__* tables (meetings, snapshots, timeline_items, conflicts, deltas, action_items, thread_excerpts, trips, trip_legs, ooo_windows, weather_snapshots, feedback_signals, email_anchors, meeting_attendees). packages/server/src/migrations.ts
  2. Scrub prior seeds
    Deletes any leftover agents (incl. legacy ea-travel-lens rows), workflows, routines from a previous install. Tasks owned by old agents get unassigned, not deleted. lifecycle.ts → scrubEaSeeds
  3. Seed workflows
    Five workflow rows: EA Sync Calendar, EA Scan Meeting Threads, EA Immediate Sync on Connect, EA Compose, EA Weekly Reflection, EA Travel Agent on triage. Each is a tiny pipeline of trigger → tool blocks.
  4. Seed agents (if Claude runtime exists)
    Inserts two rows into agents with roles executive-assistant and ea-travel-agent. If the tenant has no Claude runtime, this step skips gracefully — re-install once the runtime exists. lifecycle.ts → seedAgent
  5. Seed routines
    Four cron entries pointed at the workflows above: sync calendar every 15 min, scan threads every 15 min, compose every 15 min (hash-guarded), weekly reflection Mondays 06:30. All start active; the sync tools fail-soft until Google connects.
  6. Immediate calendar sync
    Inline calendar.sync_window call. Doesn't wait for the next cron tick — pulls 7 days of events into the meetings table right now. Fail-soft when Google isn't connected.
  7. Kick off prep briefs + first compose task
    compose.prepare_meeting_briefs creates an async agent task to write 80-word prep briefs for upcoming meetings. compose.maybe_create_task creates the first morning-dossier compose task. Both are queued, not run synchronously — the agent runtime picks them up moments later.
  8. One-shot endsAt backfill + conflict scan v0.4.12+
    Backfills timeline_items.endsAt from the parent meeting/ooo for any pre-existing snapshot whose rows came in with null end times. Then calls conflicts.recompute_for_date so the AttentionStrip reflects overlaps on the existing snapshot immediately. lifecycle.ts onInstall (tail)
Why two compose tasks? prepare_meeting_briefs writes per-meeting briefs only (faster, ~30s typical). maybe_create_task writes the full daily dossier (snapshot + narrative_brief + timeline_items + action_items + thread_excerpts). They run in parallel.

Scene 2 — Calendar arrives (first sync)

SCENE · 02 Google Calendar events become rows in the meetings table. No LLM

Pure data flow. Triggered three ways: the inline install call, the every-15-min cron, and the immediate-on-connect workflow that fires when any connector completes OAuth.

Google Calendar API │ list_events (next 14 days) ▼ calendar.sync_window ───── packages/server/src/tools/calendar.ts │ per event: ▼ meetings.upsert_from_calendar │ upserts on (tenant_id, google_event_id) │ if existing && (title or times changed): │ └─► detectMeetingDelta │ └─► writes a deltas row │ └─► recomputeConflictsForSnapshot (legacy side-effect path) ▼ meetings table ◄── this is what the UI's FocusCards read live

The brand-new bypass. If existing.length === 0 (a meeting the agent has never seen), isChanged is false and detectMeetingDelta doesn't fire. That's fine — brand new meetings are caught by the next compose tick (the compose hash includes meeting ids, so a new meeting changes the hash, wakes the agent, who writes a new snapshot, whose timeline_items.create call fires the conflict scan). See Scene 4.

Scene 3 — The first compose (agent wakes)

SCENE · 03 The compose task lands in the agent's queue. The runtime picks it up and feeds SKILL.md + the task description to Claude. LLM-driven (~30–90s)

The agent reads its SKILL.md and steps through it. Numbers below match the file.

  1. 0. Sync the calendar (idempotent)
    Calls executive-assistant.calendar.sync_window with days: 14. Cheap if the install just synced — the upsert is a no-op when nothing changed.
  2. 1. Fetch today's structured signal
    Three tool calls: meetings.list, ooo.list, trips.list. Bounds: today 00:00 → 23:59 local.
  3. 2. Resolve location, fetch weather
    Reads david.home_location from memory (or the trip leg location if traveling). Calls weather.fetch_for_date. Skips silently if memory has no location yet.
  4. 3. Per-meeting deep read
    For each meeting from step 1, calls meetings.get — pulls attendees, description (agenda), conference link.
  5. 4. Recall soft context
    memory.recall on david.voice, david.cadence, people.<email> per external attendee, company.<domain>. Shape the prose; never invent facts.
  6. 5. Per-meeting prep brief
    For each meeting, composes ≤80 words and calls meetings.set_brief. Opens with what the meeting is for (from the agenda), adds prep cues. No invented context.
  7. 6. Verbatim thread excerpts
    Up to two per meeting via thread_excerpts.create. Verbatim only — never paraphrased.
  8. 7. Action items
    Open items inferred from email threads via action_items.create. owedBy is either "david" or the external email.
  9. 8. Day's narrative brief
    Single paragraph, ≤80 words, in David's voice. Mentions weather only when notable. Writes via snapshots.create which also computes the state hash for the cron gate.
  10. 9. Timeline items
    For each meeting + trip leg + OOO, calls timeline_items.create. The server-side handler denormalises startsAt and endsAt from the entity — the agent doesn't have to think about these fields. This is also where the conflict scan fires (see Scene 4).
  11. 10. Reflection (optional writes)
    Reads recent feedback_signals. Writes contextual memory only when a pattern is confirmed (2+ rows in the same direction). Trivial things stay silent.
  12. 11. Mark task done
    framework.tasks.patch with status: done.

Scene 4 — Conflict scan (server-side, no LLM)

SCENE · 04 Every timeline_items.create fires recomputeConflictsForSnapshot as a server-side side-effect. No LLM · v0.4.11+

Conflict detection is pure pairwise overlap math. There's no judgment to make. The server runs it without asking the agent — eliminating the historical fragility where the LLM had to remember to call a "scan" tool.

timeline_items.create ────── packages/server/src/tools/timeline_items.ts │ 1. tenant guard via snapshot ownership │ 2. resolve startsAt/endsAt from input OR the underlying │ meeting / ooo_window / trip_leg row (denormalisation) │ 3. INSERT row │ 4. SIDE-EFFECT (best-effort, try/catch): ▼ recomputeConflictsForSnapshot(db, tenantId, snapshotId) │ select all timeline_items in this snapshot │ filter to those with non-null startsAt + endsAt │ O(n²) pairwise: x.startsAt < y.endsAt && y.startsAt < x.endsAt │ UPSERT each overlapping pair into conflicts │ ON CONFLICT (snapshot_id, a_id, b_id) DO UPDATE │ auto_resolved for any pair that no longer overlaps ▼ conflicts table ◄── this is what the AttentionStrip reads

The function fires N times per compose (one per timeline_item insert) — wasteful but cheap (n² over a day's items is tiny) and the upsert is idempotent. The last fire sees the full set, which is the one that matters.

Why server-side

Pure math. The LLM adds no value here. By moving the trigger out of SKILL.md, the conflict scan runs even when the agent crashes mid-compose. Resilience.

Other entry points

recompute_for_snapshot (agent / explicit tool), recompute_for_date (install hook, UI on-demand), and the legacy detect*Delta path on calendar mutations all call the same function. Belt-and-suspenders.

Scene 5 — Travel ingestion (event-driven)

SCENE · 05 An airline confirmation lands in the inbox. The framework triages it. The EA travel agent reads it. LLM-driven · event-keyed

Separate from the cron path entirely. Wakes on a single event: triage.classified, emitted by the framework's generic triage agent once per inbox item.

Gmail message → framework.inbox → triage agent │ emits triage.classified ▼ EA Travel Agent on triage │ creates a task assigned to │ the ea-travel-agent role ▼ ea-travel-agent wakes │ reads inbox item (subject, body) │ checks metadata.eaTravelLens (dedup) │ decides: travel? yes/no │ if travel: │ LLM extracts kind + fields │ └─► trip_legs.reconcile_from_email │ (PNR-based dedup) │ └─► email_anchors.bind │ writes metadata.eaTravelLens ▼ trip_legs table

The agent is intentionally narrow: one inbox item per wake, one decision ("travel?"), one downstream call. The shared metadata.eaTravelLens JSON on the inbox item is the persistence contract (kept as-is on the rename to avoid orphaning prior installs).

Scene 6 — The user reads /executive

SCENE · 06 The dossier surface assembles itself from a handful of independent React Query hooks. No LLM · pure read

Each surface piece is its own query. They poll at staggered intervals, so a server-side update lands in the UI within ~60s without manual refresh.

On-demand conflict scan. When the user navigates to a different day via WeekStrip, Today.tsx fires conflicts.recompute_for_date once per (date, snapshot id) session. This catches the case where the agent's step 9 finished but conflicts didn't land (and is a no-op safety net otherwise — the recompute is idempotent).

Scene 7 — Continuous operation

SCENE · 07 Every 15 minutes, the framework fires the routines. The hash gate decides whether the agent wakes. Periodic · hash-guarded

The contract: the agent only wakes when something material has shifted. The hash is computed over today's meetings, today's deltas and conflicts, OOO windows, and the weather snapshot's fetched_at.

cron (every 15 min) ──► EA Compose workflow │ ▼ compose.maybe_create_task │ computeStateHash(tenant, today) │ ┌─────────────────┴────────────────┐ ▼ ▼ hash == prior.stateHash hash != prior.stateHash return { skipped: true } framework.tasks.create └► agent wakes (Scene 3)

Parallel routines:

  • EA Sync Calendar — every 15 min, deterministic. Picks up Google edits within a window.
  • EA Scan Meeting Threads — every 15 min, deterministic. Anchors known-meeting threads in email_anchors. No LLM.
  • EA Compose — every 15 min, hash-guarded LLM wake (above).
  • EA Weekly Reflection — Mondays 06:30 local. Always wakes (no hash gate).
  • EA Travel Agent on triage — event-driven; not on a routine.

Scene 8 — Memory & reflection

SCENE · 08 The agent learns David's preferences from how he edits the dossier. LLM-driven · slow loop

Every user edit on a meeting brief, every conflict resolution, every acknowledged delta becomes a row in feedback_signals. The agent reads these in two passes:

End-of-run (every compose)

SKILL.md step 10. Reads recent signals, looks for a confirmed pattern (2+ rows same direction), maybe writes david.voice or people.<email> via memory.remember. Bar: "I'd defend this in front of David."

Weekly deep reflection

Monday 06:30. Reads the last 7 days. Looks for trends a single-day reflection can't see (e.g. five dismissed action items from one company → "agent is too eager extracting action items from Acme threads"). Updates contextual memory.

Memory tags are namespaced strings: david.voice, david.cadence, david.lodging.preferences, people.<email>, company.<domain>. The agent reads them at the top of every compose (Scene 3, step 4) to shape prose — but the memories never supply facts; only tone and emphasis.

The data model — at a glance

Thirteen module-owned tables. Most are obvious; a few have non-obvious roles worth knowing.

Table What it holds Notable
meetings One row per Google Calendar event the agent cares about. Survives across days; the per-meeting brief lives here.
meeting_attendees External attendees with optional bio. Replaced on every upsert; bios persist via memory.
snapshots One composed dossier per (tenant, date). Stores state_hash for the cron gate; superseded rows are kept for history.
timeline_items Pointers from a snapshot to the entities for that day. Polymorphic kind: meeting / trip_leg / ooo. startsAt/endsAt are denormalised copies of the entity's times.
conflicts Pairwise overlaps between two timeline_items. Canonicalised (a < b). Resolution status: unresolved / acknowledged / resolved_by_user / auto_resolved.
deltas Notable shifts since a snapshot was composed. Includes "moved 30 min later", "flight delayed", etc.
action_items Open items inferred from meeting threads. Dismissal writes a feedback signal for the reflection step.
thread_excerpts Verbatim email quotes attached to a meeting. Never paraphrased; sender + timestamp preserved.
trips One row per logical trip (with multiple legs). Status: planned / in_progress / completed / cancelled.
trip_legs Flight, hotel, or ground transport segments. PNR-based dedup; built incrementally from emails.
ooo_windows Out-of-office periods. Source = calendar event or auto-responder email.
weather_snapshots Daily Open-Meteo pull per location. Driven by david.home_location memory or active trip leg.
feedback_signals Every meaningful user interaction with the dossier. Fuels the reflection loop. Telemetry-only signals don't trigger writes by themselves.
email_anchors Binds a Gmail message id to an EA entity. Idempotent on (tenant, message_id). Two writers: bulk anchor + ea-travel-agent.

Failure modes — what each scene can do wrong

Knowing where each scene can break shortens debugging time. Pair this with OPERATIONS.md for the runbook side.

Scene Failure mode What it looks like Recovery
1 Install No Claude runtime on tenant Agents not seeded; everything downstream skips. Console warns once. Provision runtime, re-install module.
2 Calendar Google not connected sync_window returns ok:true with skipped: google_not_connected. meetings stays empty. Complete OAuth → "Immediate Sync on Connect" workflow fires within seconds.
3 Compose Agent task stalls / fails Day brief stays stale (or empty). agent_runs shows failure. Read the error; usually a memory provider issue or a tool transient.
4 Conflict timeline_items have null endsAt conflicts.list returns empty even with overlapping events. Fixed in v0.4.12 — server denormalisation + one-shot backfill on install.
5 Travel Triage didn't classify, or metadata.eaTravelLens already present Inbox item has no trip_leg. Agent never woke. Check triage's metadata.triage first. If absent, triage didn't run.
6 UI Snapshot has narrative_brief, but UI shows empty Likely browser cache or HMR stuck. Hard-refresh. Cmd+Shift+R; check Network tab for the response.
7 Cron Hash never changes → agent never wakes Day brief stays at install-time content. No new compose tasks created. An edit to any meeting, OOO, or weather refresh will change the hash. Or invalidate state_hash manually.
8 Memory MemoryProvider unavailable Agent composes from structured data alone (no voice tuning). Skips reflection. Configure a memory provider; next compose picks it up.
