EA Calendar Autonomy — Path A

How to give the Executive Assistant its own calendar client without breaking BoringOS

The simple version

Right now EA is like a restaurant that can't cook.
Every time it needs food, it shouts an order at the kitchen next door (BoringOS's Google connector). If that kitchen rearranges its menu, EA's order comes back wrong — or not at all.

Path A gives EA its own kitchen.
EA still buys groceries from the same store (Google's API). And it still needs one thing from the neighbour — the key to the store (an OAuth token). But once it has the key, it shops, cooks, and plates everything itself. The neighbour's menu changes are irrelevant.

How it works, step by step

B

BoringOS adds one method to ModuleFactoryDeps

getConnectorToken(kind: "google") — returns a fresh, valid access token for the tenant that installed the module. BoringOS handles token refresh internally. This is the only BoringOS change needed.

↓
1

EA calls getConnectorToken at the start of a sync

EA's own EaCalendarClient class asks BoringOS: "give me the Google token." If the token is expired, BoringOS refreshes it before returning. EA doesn't know or care.

↓
2

EA fetches events directly from Google Calendar API

EA sends a plain fetch() to https://www.googleapis.com/calendar/v3/calendars/primary/events with the token as a Bearer header. EA controls exactly which parameters it sends — timeMin, timeMax, syncToken, whatever it needs.

↓
3

EA parses the response with its own types

EA defines GoogleCalendarEvent exactly the way it wants. No guessing what shape BoringOS's connector happens to return today. If Google adds a new field EA wants, EA just uses it.

↓
4

EA saves a nextSyncToken per tenant

Google returns a nextSyncToken at the end of every event list. On the next sync, EA passes it back — Google only returns events that changed. This makes subsequent syncs tiny and fast. EA stores this token in its own table (executive_assistant__sync_state).

↓
5

EA upserts the events into its meetings table as before

Nothing about the rest of the pipeline changes. The meetings table stays the same. The UI hooks stay the same. Only the source of the raw events changed.

The one BoringOS change required

packages/@boringos/module-sdk/src/types.ts — add one field
export interface ModuleFactoryDeps {
  db:             unknown;
  memory?:        MemoryProvider;
  toolRegistry?:  unknown;
  // ... existing fields ...

  // NEW — lets modules get a live OAuth token for an installed connector
  getConnectorToken?: (
    kind: "google" | "slack"
  ) => Promise<{ accessToken: string } | null>;
}
BoringOS wires this up to its existing internal loadGoogleCreds + refreshOAuthToken logic — the same code the built-in Google module already uses. For EA, this is the only surface it needs from BoringOS going forward. Every other Google Calendar capability is EA's own code.

What EA gains

🔒

No more silent breakage

BoringOS can rename or restructure its Google connector however it wants. EA doesn't call it anymore.

⚡

Incremental sync (syncToken)

After the first full sync, each subsequent sync only fetches changed events — instead of up to 250 events every 15 min.

📅

Full API access

EA can query multiple calendars, filter by colour, read attendee response status — anything Google Calendar supports.

🔔

Webhooks (future)

EA can register a watch channel with Google so changes push in seconds instead of waiting for the next poll tick.

🧩

Own types, own schema

EA defines exactly what a calendar event means to it. No adapting to whatever shape BoringOS happens to return.

🗺️

Other calendars later

EA can add Outlook / Apple Calendar support independently without waiting for BoringOS connectors to appear.
