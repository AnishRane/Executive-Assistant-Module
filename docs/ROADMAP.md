# Executive Assistant — Roadmap

> Forward-looking plan for the EA module. Pairs with `PHASE_PLAN.md`
> (which is the historical record of how v1 was built). This doc is
> about where we go next, what's already in the box, and how the
> agent landscape will evolve.

## 1. Current state

The EA module ships **Daily Operations** from the master spec — the
morning dossier surface at `/executive` that shows the user's day at
a glance: meetings, travel, OOO, weather, and (once the in-flight fix
lands) scheduling conflicts.

Two agents are deployed today:

| Role id              | Priority | Built? | Wake trigger                                  | One-liner                                              |
| -------------------- | -------- | ------ | --------------------------------------------- | ------------------------------------------------------ |
| `executive-assistant`| 60       | ✅     | Cron every 15 min, hash-guarded               | Composes the daily dossier from structured signals.    |
| `ea-travel-agent`    | 50       | ✅     | `triage.classified` event (per inbox item)    | Reads travel emails → persists `trip_legs`.            |

Shipped in v0.4.9: conflict-detection trigger fix and the
`ea-travel-lens` → `ea-travel-agent` rename (see section 3 below
for the historical action list). Investigation note for the
conflict bug: `docs/conflict-detection-investigation.md`.

## 3. Renames

**Status: shipped in v0.4.9.** Section kept as a historical record
of the action so future readers know why `ea-travel-lens` references
appear in scrub lists, legacy comments, and pre-v0.4.9 agent_runs.

Only one rename was needed to bring the code in line with the spec
naming convention. The convention adopted:

  - Main agent: bare (`executive-assistant`)
  - Sub-agents: `ea-<thing>-agent`

### Action: `ea-travel-lens` → `ea-travel-agent`

Files to touch:

| File                                                                       | Change                                            |
| -------------------------------------------------------------------------- | ------------------------------------------------- |
| `packages/server/src/skills/ea-travel-lens/SKILL.md` (rename file/dir)     | `id` and `roles` in frontmatter; file path        |
| `packages/server/src/module.ts`                                            | Skill registration                                |
| `packages/server/src/lifecycle.ts`                                         | Agent seeding (find by role name)                 |
| `packages/server/dist/...` (regenerated by build)                          | Rebuilt automatically                             |
| Any test fixtures referencing the old role id                              | Search-and-replace                                |

Migration consideration: any existing tenant that has open tasks
assigned to `ea-travel-lens` role will need those tasks re-pointed.
The agent-seeding logic in `lifecycle.ts` already creates agents on
install; on rename, the install path should also alias / migrate the
old role to the new one. Worth a one-off migration script for
tenants on prior versions.

The spec itself should also be updated to formally name **Travel
Agent** as a sub-agent of Executive Assistant (currently it appears
only as an output, "Travel digest", under Daily Operations).

## 4. Agent charters

Each agent's operational scope and responsibilities. Forward-looking:
includes future agents from the spec.

### `executive-assistant` (main, built)

**Domain:** Composing the daily dossier. The user-visible "what does
my day look like" surface.

**Wake triggers:**
- Every 15 min via `compose.maybe_create_task` cron (hash-guarded —
  only wakes when state actually changed).

**Reads:** `meetings`, `meeting_attendees`, `trips`, `trip_legs`,
`ooo_windows`, `weather_snapshots`, `snapshots`, `deltas`,
`conflicts`, `feedback_signals`, plus framework.memory.

**Writes:** `snapshots` (narrative_brief), `timeline_items`,
per-meeting briefs (`meetings.brief`), `action_items`,
`thread_excerpts`; updates contextual memory at end-of-run.

**Out of scope (today):** No outbound email. No calendar edits. No
meeting joins. No content drafting beyond the day's brief. (Drafts
for conflict resolution land in v0.2 per current SKILL.md.)

**Owned tools:** `meetings.{list,get,set_brief}`,
`trips.{list,get}`, `ooo.list`, `snapshots.create`,
`timeline_items.create`, `thread_excerpts.create`,
`action_items.create`, `feedback_signals.list`,
`weather.fetch_for_date`, `calendar.sync_window`,
`memory.{recall,remember,forget}`.

### `ea-travel-agent` (built, pending rename)

**Domain:** Email-driven travel ingestion. Turns confirmation /
update emails into structured `trip_legs` rows that the main EA
consumes for the travel digest.

**Wake triggers:**
- `triage.classified` event (one per inbox item) emitted by the
  framework's `triage` agent. Idempotency-keyed on
  `metadata.eaTravelLens.processedAt`.

**Reads:** Inbox item bodies + the existing `metadata.triage`
classification. Does NOT re-classify priority.

**Writes:** `trip_legs`, `email_anchors`; updates inbox item
`metadata.eaTravelLens`.

**Out of scope:** Composing the dossier. Reasoning about meeting
overlap or schedule structure. Anything not derivable from one
inbox item at a time.

**Owned tools:** `trip_legs.reconcile_from_email`,
`email_anchors.bind`, `framework.inbox.{read,update}`.

### `ea-meetings-agent` (future, Phase 2)

**Domain:** Pre/intra/post-meeting operations for external advisor
meetings — joining, transcribing, summarizing, action-item extraction
from transcripts, board meeting docs.

**Wake triggers (planned):**
- Calendar event detection (new external-advisor meeting added).
- Scheduled at meeting start time (auto-join).
- After-meeting completion (transcript ready → summarize, extract).

**Reads (planned):** `meetings`, `meeting_attendees`, calendar API,
transcript source (Granola or platform-native), the framework's
memory for advisor context.

**Writes (planned):** `meeting_transcripts` (new table),
`action_items` (extends current source — currently email-thread
inferred, will gain transcript-inferred), board-meeting docs.

**Out of scope:** Composing the day's dossier (still main EA's job).
News, drafts, family schedules, finances. Internal/personal
meetings — only external-advisor meetings.

**Owned tools (planned):** `meeting_transcripts.upsert`,
`granola.fetch_notes`, `meetings.join_zoom`, `meetings.join_meet`,
`meetings.join_teams`, `advisors.list`, etc.

### `ea-news-drafts-agent` (future, Phase 3)

**Domain:** Community engagement — daily news digest + draft replies
for community forums (Reddit, Long Angle, etc.).

**Wake triggers (planned):**
- Scheduled daily at user-configured digest time.
- New content arrives in a monitored feed (optional).

**Reads (planned):** External feeds — Hacker News, Reddit
fatFIRE, Long Angle, Substack subscriptions.

**Writes (planned):** `news_digests` (new table), `drafts` (new
table).

**Out of scope:** Calendar/meeting concerns. Travel. Action items.

**Owned tools (planned):** `hn.fetch`, `reddit.fetch`,
`substack.fetch`, `news_digest.create`, `drafts.create`, possibly
`email.send` (or via a framework tool).

## 5. Roadmap by phase

### Phase 1 — Daily Operations (current, ~90% complete)

**Goal:** The morning dossier surface works end-to-end with full
signal coverage.

**Agents:** `executive-assistant`, `ea-travel-agent` (post-rename).

**What's shipped:** Morning digest (meetings + weather + OOO),
travel digest, per-meeting briefs, the FocusCard / LaterList / WeekStrip
UI, the install-time data seeding (v0.4.7), the FocusCard + Day Brief
skeleton-state UX (v0.4.8).

**What's left in Phase 1:**
- Conflict-detection trigger fix (in-flight brainstorm — see
  `docs/conflict-detection-investigation.md`).
- The `ea-travel-lens` → `ea-travel-agent` rename.
- Spec-update to formally name "Travel Agent" as a sub-agent.

**Done when:** A user opening `/executive` sees their day with no
silent gaps — conflicts surface when they exist, travel ingests
within minutes of receiving a confirmation email, the dossier
re-composes on schedule changes.

### Phase 2 — Meeting Operations + Meetings Agent

**Goal:** External advisor meetings become first-class — auto-joined,
transcribed, summarized; action items extracted from what was
actually said.

**New agent:** `ea-meetings-agent`.

**New surfaces:**
- Per-meeting page enrichment: transcript summary, in-meeting action
  items, post-meeting follow-up drafts (handed to Meetings Agent's
  outputs).
- External-advisor scoping — only meetings tagged as advisor meetings
  (definition TBD — see Open Questions).

**Refactors to existing code:**
- Per-meeting brief generation: today, main EA writes briefs for
  every meeting based on agenda + memory. In Phase 2, advisor briefs
  start pulling additional context from Meetings Agent (prior
  transcripts, action-item history). The brief content gets richer;
  the write path stays in main EA.
- Action item extraction expands: today purely from email threads
  inferred by main EA; in Phase 2 the dominant source becomes
  transcripts written by Meetings Agent.

**Dependencies (open, possibly blocking):**
- BoringOS connectors for Zoom / Google Meet / Teams (probably new
  modules in the framework).
- Granola integration or transcript-source equivalent.
- A definition of "external advisor" — likely a new table
  `external_advisors` or a tag on `meeting_attendees`.

**Done when:** An external advisor meeting auto-joins, produces a
transcript, surfaces a summary + action items on the meeting drawer,
and the next day's morning dossier reflects what was decided.

### Phase 3 — News & Drafts Agent

**Goal:** Community-engagement loop — surface interesting external
content daily, and draft user replies in their voice.

**New agent:** `ea-news-drafts-agent`.

**New surfaces:**
- Daily news email digest at user-configured time.
- Drafts queue (review-before-send) on the dossier surface or its
  own page.

**Dependencies (open):**
- Email-send capability (framework.email or new connector).
- Reddit / HN / Substack reader connectors.

**Done when:** The user gets a daily digest email; relevant
discussion threads surface in the dossier with one-click draft
replies ready to review.

## 6. Out of scope (separate modules)

Per the broader Personal AI Stack spec, these are top-level
assistants that are **not** part of this module:

| Spec agent                  | Why separate                                                    |
| --------------------------- | --------------------------------------------------------------- |
| Document Assistant          | Different domain (file ingestion, retrieval). Own module.       |
| Family Manager (future)     | Different user (family schedules), different data model.        |
| Personal OpEx Assistant     | Personal-finance / operations domain. Own module.               |
| Controller (future)         | Financial control. Distinct from operations.                    |

If/when they ship, they'll be peer modules to `executive-assistant`,
not sub-agents inside it.

## 7. Open design questions

Tracked here so they don't get lost between phases.

1. **Phase 2 — per-meeting brief migration.**
   When Meetings Agent's transcripts come online, do advisor briefs
   live with Meetings Agent (and the main EA reads them), or stay
   in main EA (which pulls transcript context from Meetings Agent)?
   Both are viable; decision affects Phase 2 SKILL.md split.

2. **Phase 2 — "external advisor" definition.**
   Source of truth: a new `external_advisors` table, or a tag on
   `meeting_attendees.is_external` already present? If the latter,
   "external" today means "not internal domain" — that's too loose
   for advisor scoping.

3. **Phase 2 — transcript storage.**
   New table `meeting_transcripts` keyed on `meeting_id`, or stored
   as opaque blobs on `meetings.transcript_uri`? First is queryable
   (search across transcripts), second is cheaper.

4. **Rename migration for tenants on prior versions.**
   Tenants installed before the `ea-travel-lens` → `ea-travel-agent`
   rename will have an agent row with the old role. The install
   path on the new version should detect and migrate, not duplicate.

5. **Item categorization — meeting / event / task.** *(brainstormed 2026-05-16,
   deferred)*  Today every calendar item flows through one path:
   "meeting." Reality is mixed — some items are meetings (have video
   links / agendas / external attendees), some are events (physical,
   travel, social, no prep brief needed), some are tasks (todo, no
   attendees). Proposal: server-side rules + LLM enrichment (not LLM
   fallback) tag each item, and the UI / agent treats categories
   differently. Open architectural question: single-enum category vs
   multi-facet flags. Full brainstorm context preserved in the May
   2026 conversation; resume when user signals.

6. **Localized fields for ooo.list + trips.list.** *(v0.4.14)*  Same
   pattern as `meetings.list` in v0.4.13 — add `tz` input, return
   `startsAtLocal`/`endsAtLocal`/`dayPart`/`tzAbbr`/`tzIana` per row,
   `tzError` on resolution failure. Low risk; tests already exist
   for the helper layer.

7. **LocaleChip + Settings location section.** *(v0.4.14)*  Per the
   ui-ux-pro-max design (top-right of header, city + tzAbbr, swap
   icon home↔away, popover on tap). Plus a Location section in
   Settings with current / home / override fields.

8. **david.\* → user.\* memory migration.** *(v0.4.14)*  v0.4.13
   ships SKILL.md using `user.*` tags. Tenants installed pre-v0.4.13
   may have agent-written values under `david.voice`,
   `david.cadence`, `david.home_location` — these get orphaned until
   migrated. One-shot install hook to copy + forget. Idempotent.

9. **Spec drift acknowledgement.**
   `ea-travel-lens` exists in code but not in the original spec. The
   spec should be amended to call out "Travel Agent" as a sub-agent
   alongside Meetings Agent and News & Drafts Agent. Track this as a
   spec-doc update, not a code change.
